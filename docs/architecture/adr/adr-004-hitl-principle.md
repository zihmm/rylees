# ADR-004: Human Review and Approval Before Publication

---

## Status

Accepted

## Context

The system uses a Large Language Model (LLM) to generate customer-facing release notes based on technical software changes.

While LLMs can significantly improve efficiency and readability, they may also produce inaccurate, incomplete or misleading content. Generated texts can contain incorrect assumptions, omit important information or formulate statements that do not accurately reflect the implemented changes.

Since release notes are customer-facing communication artifacts, the published content must meet high standards of correctness, clarity and professionalism.

Several alternatives were considered:

### Alternative A: Fully Automated Publication

Release notes generated by the LLM are published immediately without human intervention.

**Advantages**

- Fully automated workflow
- Minimal manual effort
- Fast publication process

**Disadvantages**

- Risk of publishing incorrect information
- No quality assurance step
- Reduced trustworthiness of published content
- Increased reputational risk

### Alternative B: Random Sampling Review

Only selected release notes are reviewed manually.

**Advantages**

- Reduced review effort
- Partial quality assurance

**Disadvantages**

- Quality issues may remain undetected
- Inconsistent review process
- Unpredictable publication quality

### Alternative C: Optional Review

Developers may decide whether to review generated content.

**Advantages**

- Flexible workflow
- Reduced manual effort in some cases

**Disadvantages**

- Inconsistent quality standards
- Increased risk of publishing inaccurate information

## Decision

All generated release notes shall require explicit human review and approval before publication.

The output produced by the LLM is considered a draft that supports the developer during the documentation process. Publication is only permitted after a developer has reviewed and approved the generated content.

The publication workflow consists of the following steps:

1. Generate release notes using the CLI Tool and LLM.
2. Store the generated release notes in the system.
3. Review and optionally edit the generated content.
4. Explicitly approve the release notes.
5. Publish the approved version for customer access.

No automated publication mechanism shall be implemented.

## Consequences

### Positive Consequences

- Improved quality and correctness of published release notes
- Reduced risk of LLM hallucinations reaching customers
- Increased trust in generated content
- Clear accountability for published information
- Opportunity to enrich or adjust generated content

### Negative Consequences

- Additional manual review effort
- Publication process is not fully automated
- Potential delays between generation and publication

### Impact on Quality Goals

| Quality Goal | Contribution |
|--------------|-------------|
| Reliability | Human validation reduces the risk of incorrect information |
| Usability | Improves clarity and relevance of customer-facing content |
| Maintainability | Allows continuous improvement of generated content |
| Trustworthiness | Ensures accountability for published release notes |
| Quality Assurance | Introduces a mandatory review gate before publication |

## Governance

The following governance rules apply:

- LLM-generated content shall always be treated as a draft.
- Publication requires explicit approval by an authorized developer.
- Approved content becomes the authoritative version visible to customers.
- Developers may modify generated content before approval.
- Automatic publication of generated release notes is prohibited.
- Future changes to the publication workflow must preserve a human approval step unless a new ADR explicitly supersedes this decision.

This ADR establishes the Human-in-the-Loop principle as a core quality assurance mechanism within the release note generation process.
